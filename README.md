# 艾森豪矩陣任務優先權決策引擎

## Proposal Report

### 動機與目標
動機與主要技術議題：人在寫待辦事項時通常是「想到什麼寫什麼」，輸入的順序非常隨機且混亂。如果只用一般的清單存檔，當任務累積到幾十個甚至上百個時，要從中找出「目前最該做哪一件事」就會變得很困難，人腦很難同時比較這麼多項目的重要性。在程式開發上，如果我們只是用傳統的陣列（Array）來存這堆亂序的資料，每次要找最優先的任務都得把整張清單掃描一遍，或是每次新增任務都要重新排序整張清單，這在處理大量任務時效能會變得很差（$O(N)$ 或 $O(N \log N)$）。

目標：開發一個能處理「隨機輸入」並「自動排序」的工具。使用者只需要負責把腦中的任務輸入進去並給出重要度與緊急程度的評分，程式會自動計算權重。底層透過 Max-Heap 資料結構，確保無論輸入順序多亂，程式都能用最快的速度（$O(1)$）直接輸出目前最重要的一件事，解決大腦與傳統清單在處理混亂資料時的低效率問題。

### 預期功能
- 任務量化輸入： 使用者可新增任務，並為其「重要性」與「緊急性」進行評分（1-10 分）。
- 自動化優先權排序： 系統根據加權公式（例如：重要性占 40%，緊急性占 60%）自動計算權重，並即時維護一個「當前最應執行」的任務清單。
- 四象限視覺化輸出： 在終端機以 ASCII 圖表形式，將任務分類顯示在「重要且緊急」、「重要不緊急」、「不重要但緊急」、「不重要不緊急」四個區塊。
- 動態狀態管理： 支援更新任務狀態（完成、修改權重、刪除），並確保排序結果同步更新。
- 資料持久化： 支援將任務清單匯出與匯入為 CSV 或 JSON 格式，確保資料不會因程式關閉而遺失。

### 使用技術
- 程式語言： Python 。
- 開發環境： VS Code / Git (GitHub)。
- 外部套件： prettytable 或 tabulate (用於終端機表格美化)。

### 評估與測試策略
為了驗證本專案的實作是否正確，且確實解決了動機中提到的傳統線性排序效能瓶頸，將採用以下策略進行評估：
1. 正確性評估 (做對的策略)：  
    - 邊界條件測試：針對極端情境進行驗證。例如：連續輸入多筆「重要度與緊急度完全相同」的任務時，系統是否能穩定運作；或者嘗試提取超過清單長度的任務時的防呆機制。
    - 結構完整性驗證：撰寫單元測試，確保在經歷連續的 Insert (新增) 與 Extract-Max (提取優先任務) 操作後，底層陣列仍嚴格遵守 Max-Heap 的樹狀結構定義。  
2. 極限壓力效能評估 (做好的策略)：  
    - 合成資料基準測試：雖然本工具的實際應用場景為個人或小組的日常待辦事項（資料量通常小於 500 筆），但為了客觀驗證 Max-Heap 在動態排序上的演算法優勢，將額外建立極限測試腳本。  
    - 效能對比指標： 腳本將自動生成 $10^5$ 筆具備隨機權重的虛擬任務。以「執行時間」為指標，量測並對比「傳統 Array 重新排序 ($O(N \log N)$)」與「Max-Heap 動態提取 ($O(1)$)」在處理這 10 萬筆無序輸入時的效能差異，以此客觀數據佐證選用此資料結構的必要性。

### 競品比較

| 工具 | 排序機制 | 動態權重（Deadline / Aging） | 資料結構透明度 |
|------|----------|------------------------------|----------------|
| Todoist / Notion | 手動拖拉排序 | 無，需使用者自行判斷 | 無 |
| Google Tasks | 建立時間排序 | 無 | 無 |
| **本專題** | Max-Heap 自動排序 | 有（Deadline 線性轉換 + Aging） | 有（視覺化 Sift-Up/Down） |

一般市售待辦工具的排序依賴使用者手動調整，無法客觀反映任務的即時優先度。本專題的差異在於：以量化評分驅動 Max-Heap 自動維護順序，並在權重因時間變動時自動重新排列，不需使用者介入。

### Prototype 預計可驗證內容

1. **核心邏輯正確性**：新增數筆重要度與緊急度各異的任務後，`next` 指令應始終回傳當下 `score` 最高的任務，且完成後 Heap 應自動重構並指向次高任務。
2. **Deadline 動態緊急度**：輸入一筆截止日期為明日的任務與一筆截止日期為兩週後的任務，前者的緊急度應顯著高於後者，並隨時間推移持續更新。
3. **Aging 防 Starvation**：新增一筆極低分任務，模擬時間經過後確認其 `score` 隨老化加成逐漸上升，最終能浮上執行順位。
4. **Heap 視覺化步驟正確性**：在 Web UI 中觀察每次新增或完成任務時，Sift-Up / Sift-Down 的節點移動路徑是否符合 Max-Heap 性質（父節點 ≥ 子節點）。
5. **資料持久化**：執行 `save` 後關閉程式，重啟後執行 `load`，確認任務清單與優先順序完整還原。
